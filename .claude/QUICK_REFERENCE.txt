  4. ROUTE IMAGE OPTIONS - COMPARISON
═════════════════════════════════════════════════════════════════════════════════

OPTION A: Mapbox Static API (Route Polyline) ⭐ RECOMMENDED
├─ Implementation: buildRouteMapUrl(geometry, token) with polyline overlay
├─ Pros:
│  ✅ Token already integrated (MapboxContext)
│  ✅ Reuses fetchMapAsBase64() function
│  ✅ Minimal new code (100-200 LOC)
│  ✅ No external dependencies
│  ✅ Proven technology (destination maps already work)
├─ Cons:
│  ❌ Polyline encoding complexity
│  ❌ 2000 char URL limit (complex routes may need decimation)
├─ Effort: 3-4 hours
├─ Cost: FREE (included in Mapbox token)
└─ File: frontend/src/utils/mapboxStaticImage.js [EXTEND]

OPTION B: Google Maps Static API
├─ Pros:
│  ✅ Familiar visual style
│  ✅ Routing already integrated
├─ Cons:
│  ❌ Needs separate API key
│  ❌ Billing per image ($2/1000)
│  ❌ Complex URL encoding
├─ Effort: 4-5 hours
├─ Cost: $2 per 1000 requests (~$2-5/month)
└─ File: app/services/google_maps_service.py [NEW ENDPOINT]

OPTION C: Canvas/Screenshot Rendering
├─ Concept: Render Mapbox GL on hidden canvas, convert to blob
├─ Pros:
│  ✅ Complete visual control
├─ Cons:
│  ❌ VERY complex (500+ LOC)
│  ❌ Performance issues (blocks main thread)
│  ❌ Browser CORS issues
├─ Effort: 8-12 hours
└─ ❌ NOT RECOMMENDED

OPTION D: Server-Side Image Generation
├─ Concept: New backend endpoint to generate images
├─ Pros:
│  ✅ Centralized, cacheable
│  ✅ Scalable
├─ Cons:
│  ❌ Additional endpoint to maintain
│  ❌ Adds export latency
├─ Effort: 5-6 hours
└─ Consider for Phase 3

  8. CONSTRAINTS & GOTCHAS
═════════════════════════════════════════════════════════════════════════════════

⚠️  URL Length: Mapbox Static API has ~2000 char limit
    → Complex routes (1000+ points) may fail
    → Solution: Simplify/decimate polyline coordinates

⚠️  Mapbox Token: May not be available
    → User-provided or environment-based
    → Gracefully fallback to text-only summaries

⚠️  Coordinate Order: GeoJSON uses [lon, lat]
    → Google Maps uses lat,lon
    → Google Maps Routes API uses {latitude, longitude}
    → ✅ Code already handles this correctly

⚠️  Day Routes May Be Incomplete: export happens during editing
    → Only fetch maps for routes that are calculated
    → Check dayRoute.geometry before attempting map generation

⚠️  Multiple Routing Backends: ORS, Mapbox, Google Maps Routes
    → Day segments may use different providers
    → Geometry format unified (GeoJSON LineString)
    → ✅ Should work transparently

Step 1: Extend mapboxStaticImage.js (50 LOC)

  export async function buildAndFetchDayRouteMap(dayRoute, token) {
    if (!token || !dayRoute?.geometry?.coordinates?.length) return undefined;
    
    // Extract start/end POIs
    const start = dayRoute.pois[0];
    const end = dayRoute.pois[dayRoute.pois.length - 1];
    
    // Build markers
    const markers = [
      `pin-s+00FF00(${start.longitude},${start.latitude})`,
      `pin-s+FF0000(${end.longitude},${end.latitude})`
    ].join(',');
    
    // Encode polyline (or use library)
    const encoded = encodePolyline(dayRoute.geometry.coordinates);
    const path = `path-3+D97706-0.7(${encoded})`;
    
    // Build URL
    const url = `${MAPBOX_STATIC_BASE}/${markers},${path}/auto/800x400@2x?access_token=${token}&padding=50`;
    
    // Fetch and convert
    return fetchMapAsBase64(url);
  }

  if (!doc.destinationId) {
